# Лабораторная 4

Постоянная ссылка на сервис: [lab4.gost.strdinc.space](https://lab4.gost.strdinc.space)

## Какие данные подаются на вход

Программа лабораторной 4 ожидает три поля:

1. `sourceBytes` — 16 байтов в hex;
2. `aMapping` — 16 индексов для построения `a` (это `K1`);
3. `bMapping` — 16 индексов для построения `b` (это `K2`).

В отличие от лабораторной 3, отдельное поле `keyBytes` здесь не требуется: мастер-ключ формируется как `K = K1 || K2`, где обе половины собираются из `sourceBytes` по двум отображениям.

## Что происходит сразу после запуска расчета

Эндпоинт `/api/lab4/run` вызывает `run_key_schedule_trace(payload)`.

В начале выполняется строгая валидация:

- `parse_hex_vector(..., "sourceBytes")` проверяет, что вход действительно содержит 16 hex-байтов;
- `parse_mapping(..., "aMapping")` проверяет 16 индексов от 0 до 15;
- `parse_mapping(..., "bMapping")` делает то же самое для второй перестановки.

После этого формируется первый блок трассировки «Исходные данные» с таблицей `source b0..b15`.

## Как строятся половины мастер-ключа `K1` и `K2`

Дальше сервис вызывает `mapping_df(source, mapping, output_name=...)` два раза:

- один раз для `aMapping` → получается вектор `a`;
- второй раз для `bMapping` → получается вектор `b`.

В отчет добавляются:

- таблица построения `a`;
- таблица построения `b`;
- текстовые итоги `K1 = a`, `K2 = b`;
- строка `K = K1 || K2`.

То есть лабораторная 4 напрямую показывает, как из исходного массива получается 256-битный мастер-ключ.

## Как генерируются константы `C1..C32`

Константы строятся в отдельном цикле по `i = 1..32`.

1. Для каждого `i` создается вектор `Vec128(i)` функцией `vec128(i)`: это 16 байтов, где первые 15 равны `00`, а последний равен `i`.
2. Затем применяется линейное преобразование: `Ci = L(Vec128(i))` через `l_fast(...)`.
3. Каждая константа сохраняется:
   - в список `constants` для дальнейшего расчета;
   - в общую таблицу `i / Vec128(i) / Ci_hex / Ci_bin`;
   - в отдельный вложенный шаг с таблицами входа и выхода.

Таким образом, в трассировке видно не только итоговый список `C1..C32`, но и подробный путь получения каждой константы.

## Как работает шаг `F[k](a, b)` в лабораторной 4

Для развертки используется компактная функция `f_compact_action(k, a, b)`.

Она выполняет стандартную схему Feistel-подобного шага:

1. `X[k](a)` через `xor_trace(k, a)`;
2. `S(X)` через `s_trace(...)`;
3. `L(S(X))` через `l_fast(...)`;
4. XOR результата с `b` (`first_result`);
5. перенос старого `a` во вторую половину (`second_result = a[:]`).

Итог шага:

- `F.first = LSX[k](a) XOR b`;
- `F.second = a`.

Отдельно важно, что даже «компактная» версия не скрывает логику: в `actions` записываются таблицы XOR, таблица S-box, сводка L и таблица финального XOR.

## Как из `K1, K2` получаются `K3..K10`

После генерации констант сервис запускает 4 блока (группы), по 8 вызовов `F` в каждом:

- блок 1 использует `C1..C8` и дает `K3, K4`;
- блок 2 использует `C9..C16` и дает `K5, K6`;
- блок 3 использует `C17..C24` и дает `K7, K8`;
- блок 4 использует `C25..C32` и дает `K9, K10`.

На каждом вызове `F` в отчете фиксируются:

- использованная константа;
- входная пара половин;
- выходная пара половин;
- вложенные подшаги `X`, `S`, `L`, `XOR`, перенос.

После каждой группы формируется таблица пары новых ключей.

## Что именно возвращает `/api/lab4/run`

После завершения развертки backend формирует:

- `actions` — полное дерево шагов для UI и экспорта;
- `summaryTableHtml` — таблица всех раундовых ключей `K1..K10`;
- `checksTableHtml` — таблица проверок корректности структуры результата.

В таблице проверок контролируются базовые инварианты:

- сгенерировано ровно 32 константы;
- получено ровно 10 раундовых ключей;
- `K1` совпадает с исходным `a`;
- `K2` совпадает с исходным `b`.

## Что видно в интерфейсе лабораторной 4

Фронтенд лабораторной 4 (`Lab4App`) отправляет payload на `/api/lab4/run`, отображает сводную таблицу `K1..K10`, отдельный блок проверок и раскрывающийся список шагов развертки.

Также доступна кнопка «Скачать в PDF», которая вызывает `/api/lab4/report/pdf` и формирует отчет на основе уже рассчитанного результата.

## Короткий итог лабораторной 4

Лабораторная 4 реализует именно развертку мастер-ключа шифра «Кузнечик»:

- из 16 исходных байтов и двух отображений строятся `K1` и `K2`;
- по ГОСТ-схеме вычисляются `C1..C32`;
- выполняются 32 применения `F` (по 8 в каждом из 4 блоков);
- получаются все раундовые ключи `K1..K10`;
- каждый этап сопровождается таблицами и промежуточными результатами.

Это соответствует целям репозитория: максимальная прозрачность вычислений и удобная пошаговая проверка вручную.
