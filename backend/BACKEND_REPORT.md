# Отчет по backend-части

Этот backend получился не просто "прослойкой между кнопкой и формулой", а центральной частью всей системы. Именно здесь принимаются пользовательские данные, приводятся к нормальному виду, по ним пошагово строится решение, формируются таблицы для интерфейса и при необходимости собирается итоговый PDF-отчет. Вся логика специально написана так, чтобы вычисления были прозрачными: без скрытых сокращений, без вызова внешних криптобиблиотек и без попыток спрятать промежуточные шаги.

По сути backend состоит из четырех частей:

- `backend/app.py` отвечает за HTTP API и за раздачу уже собранного фронтенда;
- `backend/src/services/trace_service.py` делает основную вычислительную работу и собирает пошаговую трассировку;
- `backend/src/services/report_service.py` превращает готовый результат в Markdown и PDF;
- `backend/scripts/generate_report.py` позволяет запускать ту же генерацию отчета из командной строки, без браузера.

Ниже я разберу эти файлы по-человечески: что в них лежит, как именно работают функции и что они возвращают.

## `backend/app.py`

Это входная точка backend-приложения. Здесь почти нет математики, зато именно отсюда начинается вся работа сервиса. Файл создает Flask-приложение, настраивает папку `dist` как источник статических файлов фронтенда и поднимает несколько маршрутов API.

Переменная `DIST_DIR` хранит путь к папке `backend/dist`. Именно туда Vite складывает production-сборку интерфейса. Объект `app = Flask(...)` создается так, чтобы Flask мог и API обслуживать, и готовый фронтенд раздавать как обычные статические файлы.

### `health()`

Это самый простой маршрут: `GET /api/health`.

На вход функция ничего специально не принимает, кроме самого HTTP-запроса, который обрабатывает Flask. Никаких данных из тела запроса здесь не нужно.

Делает она одну вещь: возвращает JSON `{"status": "ok"}`. Такой маршрут удобен для проверки, что контейнер поднялся, приложение запустилось и API вообще отвечает.

На выходе всегда отдается корректный JSON-ответ со статусом 200.

### `run()`

Эта функция обрабатывает `POST /api/run`. Это основной рабочий endpoint, через который фронтенд получает полное решение по введенным данным.

На вход ожидается JSON-объект с четырьмя полями:

- `sourceBytes` — 16 исходных байтов;
- `aMapping` — 16 индексов для построения вектора `a`;
- `bMapping` — 16 индексов для построения вектора `b`;
- `keyBytes` — 16 байтов ключа.

Сначала `run()` пытается достать тело запроса через `request.get_json(silent=True)`. Если тело пустое или это не JSON, функция сразу возвращает ошибку 400 с текстом, что тело запроса должно быть JSON.

Если JSON есть, дальше управление передается в `run_trace(payload)`. Именно там происходит вся тяжелая работа: разбор входных данных, перестановки, XOR, S, R, L, Feistel-шаг и сборка финальной структуры для фронтенда.

Если внутри `run_trace` возникает `ValueError`, это считается ошибкой пользовательского ввода, и функция возвращает 400. Если случается любая другая ошибка, она трактуется как внутренняя проблема сервера, и тогда возвращается 500.

На выходе в нормальном случае `run()` отдает JSON с двумя основными полями:

- `actions` — список шагов и подшагов, которые фронтенд показывает в виде выпадающих блоков;
- `summaryTableHtml` — HTML-таблица с финальной сводкой.

### `report_pdf()`

Эта функция обрабатывает `POST /api/report/pdf` и нужна для кнопки скачивания PDF из интерфейса.

На вход она принимает тот же JSON, что и `run()`: исходные байты, отображения для `a` и `b`, а также ключ.

Работает функция так:

1. пытается получить JSON из запроса;
2. если JSON отсутствует, возвращает 400;
3. вызывает `build_pdf_report(payload)`;
4. получает имя файла, Markdown-версию отчета и готовые PDF-байты;
5. отдает PDF как файл через `send_file(...)`.

Здесь есть отдельная обработка `RuntimeError`. Она нужна на случай, если сам отчет построить удалось, а вот конвертация в PDF не сработала, например из-за отсутствия `pandoc` или проблем у PDF-движка.

На выходе, если все прошло хорошо, клиент получает бинарный PDF-файл с именем в формате `gost-full-report-YYYYMMDD-HHMMSS.pdf`.

### `index()`

Маршрут `GET /` служит стартовой страницей веб-приложения.

На вход ничего пользовательского не принимает.

Функция проверяет, существует ли файл `backend/dist/index.html`. Если фронтенд еще не собран, она возвращает JSON-ошибку 500 с подсказкой, что нужно выполнить сборку через `npm install` и `npm run build`.

Если файл есть, функция просто отдает `index.html` из папки `dist`.

На выходе получается либо HTML стартовой страницы, либо понятная ошибка о том, что фронтенд еще не собран.

### `spa(path: str)`

Этот маршрут нужен для SPA-навигации и статических ресурсов. Он обслуживает `GET /<path:path>`.

На вход принимает строку `path`, которую Flask извлекает из URL.

Логика тут такая:

- если `dist/index.html` отсутствует, возвращается ошибка о несобранном фронтенде;
- если в папке `dist` есть конкретный файл по запрошенному пути, отдается именно он;
- если такого файла нет, Flask все равно возвращает `index.html`, чтобы React Router или другая SPA-логика могла сама разобрать путь уже на стороне браузера.

На выходе это либо статический файл, либо fallback на `index.html`.

### Блок `if __name__ == "__main__":`

Это обычная секция для локального запуска. Здесь читаются переменные окружения:

- `FLASK_HOST`;
- `FLASK_PORT`;
- `FLASK_DEBUG`.

После этого вызывается `app.run(...)`.

Отдельной функции тут нет, но смысл важный: в development-режиме backend можно запускать напрямую как обычный Python-скрипт.

## `backend/src/services/trace_service.py`

Это главный файл backend-логики. Если `app.py` можно считать дверью в систему, то `trace_service.py` — это вся основная "начинка". Здесь живут и вспомогательные функции форматирования, и разбор пользовательского ввода, и операции над байтами, и построение таблиц, и сборка целого дерева шагов для фронтенда.

Сначала в файле задаются константы:

- `MOD_POLY_FULL = 0x1C3` — полный вид неприводимого полинома;
- `MOD_POLY_LOW = 0xC3` — младшая часть того же полинома, которая используется при редукции;
- `PI` — таблица прямого S-box;
- `PI_INV` — обратный S-box, который вычисляется на основе `PI`;
- `L_COEFS` — коэффициенты линейного преобразования.

Дальше уже идут функции.

### `b8(value: int) -> str`

Это маленькая служебная функция для красивого отображения байта.

На вход получает целое число `value`. По смыслу ожидается байт, но функция на всякий случай маскирует его через `& 0xFF`.

На выходе возвращает строку из восьми бит, например `10010101`.

Эта функция сама по себе ничего не вычисляет из криптографии, но используется почти везде, где нужно показать число в бинарном виде.

### `poly_str(value: int, width: int = 8) -> str`

Эта функция превращает число в полиномиальную запись над GF(2).

На вход она получает:

- `value` — число, которое надо разложить по степеням `x`;
- `width` — сколько бит учитывать. По умолчанию это 8, то есть обычный байт.

Внутри число переводится в двоичную строку фиксированной длины, потом каждый установленный бит преобразуется в соответствующий член полинома.

Например, для `0x95` на выходе получится строка вроде `x^7 + x^4 + x^2 + 1`.

Если число равно нулю, функция возвращает `"0"`.

### `bytes_hex_str(arr: list[int]) -> str`

На вход принимает список байтов `arr`.

Возвращает одну строку, в которой все байты перечислены в шестнадцатеричном виде через пробел. Например: `55 65 51 33 ...`.

Эта функция нужна в основном для коротких итоговых подписей и сводок.

### `bytes_bin_str(arr: list[int]) -> str`

Работает почти так же, как предыдущая, только выводит не hex, а бинарное представление каждого байта.

На вход получает список байтов.

На выходе возвращает строку из 16 бинарных блоков по 8 бит, разделенных пробелами.

### `_normalize_tokens(value: Any, field_name: str) -> list[str]`

Это первый шаг разбора пользовательского ввода.

На вход функция получает:

- `value` — либо строку, либо список;
- `field_name` — имя поля, чтобы в сообщении об ошибке было понятно, что именно сломано.

Если пришла строка, она разбивается по пробелам и запятым. Если пришел список, каждый элемент приводится к строке. Если пришло что-то третье, выбрасывается `ValueError`.

На выходе функция возвращает список строковых токенов, с которыми уже удобно работать дальше.

### `parse_hex_byte(token: str) -> int`

Это точечный разбор одного байта.

На вход получает строку `token`, например `95`, `0x95` или `a`.

Функция убирает префикс `0x`, приводит символы к нижнему регистру и проверяет регулярным выражением, что перед нами действительно шестнадцатеричное число длиной 1 или 2 символа.

Если формат корректный, на выходе возвращается целое число от 0 до 255. Если нет, выбрасывается `ValueError`.

### `parse_hex_vector(value: Any, field_name: str) -> list[int]`

Эта функция разбирает уже не один байт, а весь 16-байтовый вектор.

На вход получает:

- `value` — строку или список с байтами;
- `field_name` — имя поля для текста ошибки.

Сначала она вызывает `_normalize_tokens(...)`, потом проверяет, что токенов ровно 16, и после этого прогоняет каждый токен через `parse_hex_byte(...)`.

На выходе получается список из 16 целых чисел.

Если байтов не 16 или какой-то байт записан неправильно, функция выбрасывает `ValueError`.

### `parse_mapping(value: Any, field_name: str) -> list[int]`

Здесь разбираются перестановки для векторов `a` и `b`.

На вход приходит строка или список, где ожидаются 16 индексов. Как и в случае с байтами, сначала данные нормализуются в список токенов.

Дальше функция:

- проверяет, что индексов ровно 16;
- пытается преобразовать их к `int`;
- убеждается, что каждый индекс лежит в диапазоне от 0 до 15.

На выходе возвращается список из 16 чисел.

Если пользователь передал не число, слишком мало/много индексов или индекс вышел за пределы диапазона, будет `ValueError`.

### `block_text(text: str) -> dict[str, str]`

Это уже не математика, а подготовка данных для фронтенда.

На вход функция получает обычную строку `text`.

Она экранирует HTML-символы и упаковывает текст в словарь вида:

```python
{"type": "text", "html": "<p>...</p>"}
```

На выходе получается готовый блок, который React может вставить на страницу.

### `df_to_html(df: pd.DataFrame) -> str`

На вход получает `pandas.DataFrame`.

На выходе возвращает HTML-таблицу, собранную через `DataFrame.to_html(...)`, без индексного столбца, с CSS-классом `trace-table`.

Это одна из ключевых функций для визуализации: вычисления сначала складываются в DataFrame, а потом уже превращаются в HTML для веб-интерфейса и PDF-отчета.

### `block_table(df: pd.DataFrame, caption: str | None = None) -> dict[str, str]`

Эта функция делает то же самое, что `block_text`, только для таблиц.

На вход получает:

- `df` — готовую таблицу;
- `caption` — необязательный заголовок над таблицей.

Сначала DataFrame превращается в HTML через `df_to_html(...)`. Если подпись передана, она добавляется перед таблицей как отдельный `<div class='table-caption'>...`.

На выходе функция возвращает словарь вида:

```python
{"type": "table", "html": "..."}
```

### `node(title: str, blocks: list[...] | None = None, steps: list[...] | None = None) -> dict[str, Any]`

Это базовый строительный блок всей трассировки.

На вход функция принимает:

- `title` — заголовок шага;
- `blocks` — список текстовых и табличных блоков;
- `steps` — список дочерних шагов.

На выходе возвращается словарь с полями `title`, `blocks`, `steps`. Именно такую древовидную структуру потом получает фронтенд и отображает в виде вложенных выпадающих окон.

### `vector_df(arr: list[int]) -> pd.DataFrame`

На вход принимает вектор из 16 байтов.

Функция собирает таблицу, где для каждого элемента показываются:

- индекс;
- hex;
- бинарное представление;
- полиномиальная форма.

На выходе возвращается DataFrame. Его удобно использовать в самом начале, когда нужно показать исходный вектор или просто разложить текущий набор байтов "по полочкам".

### `mapping_df(source: list[int], mapping: list[int], output_name: str) -> tuple[list[int], pd.DataFrame]`

Эта функция строит новый вектор по таблице индексов.

На вход получает:

- `source` — исходный список байтов;
- `mapping` — список индексов, откуда брать элементы;
- `output_name` — имя целевого вектора, например `a` или `b`.

Для каждого нового индекса функция выбирает соответствующий элемент из `source`, добавляет его в результирующий список и одновременно формирует строку таблицы с пояснением, какой именно байт был взят.

На выходе возвращается пара:

- готовый список байтов нового вектора;
- DataFrame с подробной таблицей соответствий.

### `xor_trace(left: list[int], right: list[int]) -> tuple[list[int], pd.DataFrame]`

Это трассируемый побайтовый XOR двух векторов.

На вход получает два списка одинаковой длины, обычно по 16 байтов.

Для каждой позиции функция считает:

- левый байт;
- правый байт;
- двоичную запись обоих;
- результат XOR в бинарном и hex-виде;
- полиномиальный вид результата.

На выходе возвращается:

- список байтов результата XOR;
- DataFrame с полной расшифровкой по всем позициям.

### `s_trace(arr: list[int], inverse: bool = False) -> tuple[list[int], pd.DataFrame]`

Эта функция делает S-преобразование или обратное S-преобразование с полной таблицей.

На вход получает:

- `arr` — вектор байтов;
- `inverse` — флаг, который определяет, использовать `PI` или `PI_INV`.

Для каждого байта показывается, что было на входе, что стало на выходе, как это выглядит в hex, bin и полиномиальной форме.

На выходе — стандартная пара:

- новый список байтов после подстановки;
- DataFrame с подробной таблицей замены.

### `gf_mul_trace(a: int, b: int) -> tuple[int, pd.DataFrame]`

Это одна из самых важных функций в файле, потому что именно она показывает умножение в GF(2^8) не как "черный ящик", а по шагам.

На вход получает два байта `a` и `b`.

Дальше функция проходит восемь итераций двоичного умножения. На каждом шаге она фиксирует:

- текущее состояние `a`;
- текущее состояние `b`;
- младший бит `b`;
- аккумулятор до и после XOR;
- результат сдвига `a << 1`;
- был ли старший бит;
- как именно выглядела редукция по модулю;
- каким будет `a` на следующем шаге;
- каким будет `b` после сдвига вправо.

На выходе возвращаются:

- итоговый байт-произведение;
- DataFrame с восемью строками побитового процесса.

### `gf_mul_fast(a: int, b: int) -> int`

Это быстрая версия того же умножения, но уже без подробной трассировки.

На вход получает два байта.

На выходе отдает один байт — результат умножения в поле GF(2^8).

По смыслу эта функция нужна не для показа пользователю, а для внутренних проверок и ускоренных вычислений там, где промежуточная таблица не нужна.

### `l_trace(arr: list[int], include_mul_bit_steps: bool) -> tuple[int, list[dict[str, str]], list[dict[str, Any]]]`

Эта функция вычисляет `l(a15..a0)` и одновременно готовит все материалы, чтобы это вычисление можно было красиво показать на сайте.

На вход принимает:

- `arr` — 16-байтовый вектор;
- `include_mul_bit_steps` — флаг, нужно ли раскрывать каждое умножение еще и по внутренним битовым шагам.

Внутри для каждой пары "коэффициент из `L_COEFS` + соответствующий байт вектора" вызывается `gf_mul_trace(...)`. Результаты умножений собираются в таблицу произведений, потом отдельно строится таблица последовательного XOR этих 16 произведений.

На выходе функция возвращает сразу три вещи:

- итоговое значение `l`, то есть один байт;
- список HTML-блоков с объяснениями и таблицами;
- список вложенных шагов с детальной расшифровкой каждого умножения, если `include_mul_bit_steps=True`.

По сути это "полуфабрикат" для `R` и `R^-1`.

### `r_step(arr: list[int], include_mul_bit_steps: bool) -> tuple[list[int], list[dict[str, Any]]]`

Эта функция выполняет одно преобразование `R(a)`.

На вход получает исходный вектор `arr` и флаг, надо ли раскрывать внутренности умножений при вычислении `l`.

Сначала вызывается `l_trace(...)`, затем новый вектор собирается по правилу:

- на нулевую позицию ставится `l(...)`;
- остальные 15 байтов сдвигаются вправо.

На выходе возвращаются:

- новый вектор после `R`;
- список шагов для фронтенда.

Шаги включают входной вектор, подробный расчет `l(...)` и таблицу, показывающую, откуда взялся каждый байт результирующего вектора.

### `r_inv_step(arr: list[int], include_mul_bit_steps: bool) -> tuple[list[int], list[dict[str, Any]]]`

Это обратное к предыдущей функции преобразование `R^-1`.

На вход получает:

- вектор `arr`;
- флаг детализации умножений.

Сначала формируется промежуточный `shifted = a1..a15 || a0`, затем для него считается `l(shifted)`, после чего результирующий вектор собирается как `a1..a15 || l(shifted)`.

На выходе функция тоже возвращает две вещи:

- новый вектор;
- список шагов и таблиц для интерфейса.

Структурно здесь шагов больше, чем у `r_step`, потому что дополнительно показывается само построение `shifted`.

### `r_fast(arr: list[int]) -> list[int]`

Это короткая, нетрассируемая версия `R`.

На вход получает вектор из 16 байтов.

Внутри быстро считает XOR всех произведений коэффициентов на элементы вектора через `gf_mul_fast(...)`, а потом возвращает новый список:

- первый байт — получившийся `l`;
- остальные — первые 15 байтов старого массива.

На выходе — список из 16 байтов.

### `r_inv_fast(arr: list[int]) -> list[int]`

Это быстрая версия `R^-1`.

На вход принимает 16-байтовый вектор, внутри считает `l` для циклически сдвинутого варианта и на выходе возвращает восстановленный список байтов.

Используется в основном для проверок обратимости.

### `l_fast(arr: list[int]) -> list[int]`

На вход получает вектор `arr`.

Функция 16 раз подряд применяет `r_fast(...)`, тем самым реализуя `L(a) = R^16(a)`.

На выходе возвращается новый 16-байтовый список.

### `l_inv_fast(arr: list[int]) -> list[int]`

Полный аналог предыдущей функции, только для обратного преобразования.

На вход получает 16-байтовый вектор, 16 раз применяет `r_inv_fast(...)` и возвращает результат.

### `s_fast(arr: list[int], inverse: bool = False) -> list[int]`

Это быстрый вариант S-преобразования без таблицы.

На вход принимает:

- список байтов;
- флаг `inverse`.

На выходе возвращает новый список, где каждый байт заменен по прямому или обратному S-box.

В проекте функция в основном нужна для лаконичных проверок, когда уже не требуется строить красивую визуализацию.

### `l_action(arr: list[int], inverse: bool = False) -> tuple[list[int], dict[str, Any]]`

Эта функция поднимает уровень абстракции выше и собирает уже не просто один шаг, а целое действие `L(a)` или `L^-1(a)` для интерфейса.

На вход получает:

- вектор `arr`;
- флаг `inverse`, который переключает прямой и обратный режим.

Дальше она 16 раз подряд вызывает либо `r_step(...)`, либо `r_inv_step(...)`. После каждого раунда сохраняется:

- состояние до;
- состояние после;
- вложенные шаги конкретного раунда.

На выходе возвращается:

- итоговый вектор после 16 раундов;
- один большой узел `node(...)`, в котором уже лежат:
  - краткое объяснение;
  - таблица истории состояний по всем 16 раундам;
  - итоговая строка;
  - список из 16 вложенных подшагов.

Именно благодаря этой функции на фронте `L` и `L^-1` можно показывать как отдельные большие блоки, внутри которых лежат раунды, а внутри раундов — еще более мелкие вычисления.

### `f_action(k: list[int], a: list[int], b: list[int]) -> tuple[tuple[list[int], list[int]], dict[str, Any]]`

Эта функция реализует один шаг Фейстеля:

`F[k](a, b) = (LSX[k](a) XOR b, a)`

На вход получает три 16-байтовых вектора:

- `k` — ключ;
- `a` — первый входной блок;
- `b` — второй входной блок.

Внутри работа идет по цепочке:

1. считается `X[k](a)` через `xor_trace(...)`;
2. результат проходит через `s_trace(...)`;
3. затем к нему применяется `l_action(...)`;
4. результат `LSX` XOR-ится с `b`;
5. вторым выходом просто становится исходный `a`.

На выходе функция возвращает:

- пару итоговых векторов `(first_result, second_result)`;
- большой узел действия для фронтенда, где каждый из пяти этапов оформлен как отдельный подпункт.

### `run_trace(payload: dict[str, Any]) -> dict[str, Any]`

Это главная функция всего backend-вычисления.

На вход она получает словарь `payload`, который приходит либо из веб-интерфейса, либо из CLI-скрипта. В словаре ожидаются поля:

- `sourceBytes`;
- `aMapping`;
- `bMapping`;
- `keyBytes`.

Сначала входные данные проходят через `parse_hex_vector(...)` и `parse_mapping(...)`. После этого backend получает уже нормальные Python-списки с числами.

Дальше функция шаг за шагом строит полное решение:

1. показывает исходные `source` и `key`;
2. строит вектор `a` по индексам;
3. строит вектор `b` по индексам;
4. считает `X[k](a)`;
5. считает `S(a)`;
6. считает `S^-1(a)`;
7. считает `R(a)` вместе с вложенными шагами;
8. считает `R^-1(a)`;
9. считает `L(a)`;
10. считает `L^-1(a)`;
11. считает `F[k](a, b)`;
12. отдельно проверяет обратимость `S`, `R` и `L`;
13. формирует финальную сводную таблицу.

На выходе `run_trace(...)` возвращает словарь из двух частей:

- `actions` — полное дерево шагов для веб-интерфейса;
- `summaryTableHtml` — HTML финальной таблицы для отдельного блока "Финальная сводка".

Если где-то на этапе разбора входных данных выявляется ошибка, функция не маскирует проблему, а выбрасывает `ValueError`, который уже выше ловится в `app.py`.

## `backend/src/services/report_service.py`

Если `trace_service.py` отвечает за сами вычисления и визуальные блоки, то `report_service.py` занимается упаковкой готового результата в отчет. Он не пересчитывает математику заново, а берет уже подготовленную структуру и переводит ее сначала в Markdown, а потом в PDF.

### `strip_simple_tags(html_text: str) -> str`

На вход функция получает небольшой HTML-фрагмент.

Она удаляет из него теги регулярным выражением, затем декодирует HTML-сущности через `html.unescape(...)` и убирает лишние пробелы по краям.

На выходе возвращается обычный текст без HTML-разметки.

Эта функция нужна для случаев, когда блок на сайте хранится как HTML, а в Markdown-отчете нужен уже просто читабельный текст.

### `split_caption_and_table(block_html: str) -> tuple[str, str]`

На вход подается HTML-строка табличного блока.

Функция пытается отдельно вытащить:

- подпись таблицы из `<div class='table-caption'>...</div>`;
- сам HTML-код таблицы `<table>...</table>`.

На выходе возвращается пара:

- текстовая подпись;
- HTML таблицы.

Это нужно, чтобы в Markdown-отчете подпись не потерялась и не смешалась с телом таблицы.

### `render_block_markdown(block: dict[str, str]) -> list[str]`

На вход приходит один frontend-блок, то есть словарь с полями `type` и `html`.

Если блок текстовый, функция очищает HTML и возвращает список строк Markdown-текста. Если блок табличный, она отделяет подпись от таблицы и тоже возвращает список строк, который потом удобно вставить в итоговый документ.

На выходе — список строк, а не одна строка. Это сделано специально, чтобы потом было удобно собирать большой Markdown-файл по частям.

### `render_node_markdown(node: dict[str, Any], level: int) -> list[str]`

На вход получает один узел трассировки `node` и число `level`, которое задает глубину заголовка в Markdown.

Функция:

- ставит заголовок нужного уровня;
- добавляет все текстовые и табличные блоки из `blocks`;
- рекурсивно проходит по `steps`, если внутри есть дочерние шаги.

На выходе снова формируется список строк. Это уже почти готовый Markdown-фрагмент для одного действия или поддействия.

### `build_markdown_report(result: dict[str, Any], payload: dict[str, Any]) -> str`

Это главная функция сборки Markdown-отчета.

На вход получает:

- `result` — то, что вернул `run_trace(...)`;
- `payload` — исходный пользовательский запрос, чтобы его тоже включить в отчет.

Сначала добавляется шапка отчета с датой и исходным JSON. Потом все действия из `result["actions"]` последовательно прогоняются через `render_node_markdown(...)`.

На выходе получается один большой Markdown-текст, готовый либо к сохранению в `.md`, либо к конвертации в PDF.

### `markdown_to_pdf_bytes(markdown: str) -> bytes`

Эта функция отвечает за переход от Markdown к настоящему PDF-файлу.

На вход получает строку Markdown.

Дальше работа идет так:

1. создается временная папка;
2. туда записывается `report.md`;
3. туда же записывается компактный CSS для печатной версии;
4. запускается `pandoc` с движком `weasyprint`;
5. готовый `report.pdf` читается обратно в память.

CSS здесь важен сам по себе: именно он задает небольшие поля страницы, уменьшенные размеры шрифтов и аккуратные таблицы с автоматической шириной колонок.

На выходе функция возвращает байты PDF-файла.

Если `pandoc` не найден, выбрасывается `RuntimeError` с понятным текстом. Если `pandoc` запускается, но сборка падает, выбрасывается другой `RuntimeError`, уже с текстом ошибки от внешней команды.

### `build_pdf_report(payload: dict[str, Any]) -> tuple[str, str, bytes]`

Это верхнеуровневая функция PDF-экспорта.

На вход получает тот же `payload`, что и API `/api/run`.

Дальше внутри последовательно вызывает:

- `run_trace(...)` — чтобы посчитать решение;
- `build_markdown_report(...)` — чтобы собрать Markdown;
- `markdown_to_pdf_bytes(...)` — чтобы превратить его в PDF.

На выходе возвращает тройку:

- имя итогового PDF-файла;
- Markdown-версию отчета;
- байты PDF.

Такой интерфейс удобен тем, что одна и та же функция сразу подходит и для веб-ручки, и для возможного повторного использования в других сценариях.

## `backend/scripts/generate_report.py`

Этот файл уже не участвует в обработке веб-запросов напрямую, но он полезен как отдельный инструмент. С его помощью можно создать отчет по тем же правилам, что и через сайт, только из консоли.

В начале файла вычисляются `ROOT_DIR` и `BACKEND_DIR`, после чего путь к backend добавляется в `sys.path`. Это нужно затем, чтобы из обычного скрипта можно было импортировать сервисы `trace_service` и `report_service`.

### `read_payload_from_args(args: argparse.Namespace) -> dict[str, Any]`

Эта функция собирает входные данные для отчета.

На вход получает уже разобранные аргументы командной строки `args`.

Дальше у нее три режима работы:

1. если указан `--payload-json`, она читает JSON-файл;
2. если переданы все нужные параметры напрямую (`--source-bytes`, `--a-mapping`, `--b-mapping`, `--key-bytes`), собирает словарь из них;
3. если ни то ни другое не выполнено, переходит в интерактивный режим и спрашивает значения у пользователя через `input(...)`.

На выходе функция всегда возвращает словарь формата:

```python
{
    "sourceBytes": ...,
    "aMapping": ...,
    "bMapping": ...,
    "keyBytes": ...,
}
```

Если путь к JSON-файлу неверный, выбрасывается `FileNotFoundError`.

### `parse_args() -> argparse.Namespace`

На вход ничего не принимает: она просто читает аргументы командной строки через `argparse`.

Внутри описываются все поддерживаемые опции:

- `--payload-json`;
- `--source-bytes`;
- `--a-mapping`;
- `--b-mapping`;
- `--key-bytes`;
- `--output-dir`;
- `--name`;
- `--skip-pdf`.

На выходе возвращается объект `argparse.Namespace`, который потом использует `main()`.

### `main() -> None`

Это основная функция CLI-скрипта.

На вход параметров явно не получает, потому что сначала сама вызывает `parse_args()`.

Дальше последовательность такая:

1. разобрать аргументы;
2. собрать `payload` через `read_payload_from_args(...)`;
3. вычислить решение через `run_trace(...)`;
4. собрать Markdown через `build_markdown_report(...)`;
5. создать каталог для отчетов;
6. сохранить `.md`;
7. если не указан `--skip-pdf`, попробовать собрать `.pdf`.

На выходе функция ничего не возвращает. Ее задача — создать файлы на диске и вывести в консоль понятные сообщения о результате.

Если PDF по какой-то причине не удалось построить, это не считается фатальной ошибкой для всего сценария: `.md` уже остается сохраненным, а пользователю просто выводится объяснение, что PDF можно будет собрать позже.

## Как все это работает вместе

Если посмотреть на backend целиком, поток данных получается довольно прямой.

Пользователь вводит исходные байты, индексы для `a`, индексы для `b` и ключ. Эти данные прилетают в `app.py`, оттуда уходят в `run_trace(...)`, где проходят полный цикл разбора и вычислений. Дальше из этого рождается дерево шагов `actions`, где каждый узел уже готов к отображению в интерфейсе. Текстовые пояснения приходят как HTML-блоки, таблицы формируются из DataFrame и тоже отдаются как HTML.

Если нужен не только просмотр на сайте, но и файл для скачивания, backend берет тот же результат и через `report_service.py` собирает из него Markdown-отчет, а затем печатную PDF-версию. За счет этого веб-интерфейс и экспорт не расходятся между собой: и там и там используется одна и та же вычислительная логика.

В результате backend решает сразу несколько задач:

- проверяет корректность ввода;
- выполняет все преобразования Кузнечика вручную;
- показывает промежуточные шаги без потерь;
- превращает вычисления в таблицы и текстовые блоки;
- отдает результат и как JSON для сайта, и как PDF для скачивания.

И это, пожалуй, его главная сильная сторона: не просто "посчитать ответ", а сделать так, чтобы было видно, откуда этот ответ вообще взялся.
