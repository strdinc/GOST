## Какие данные подаются на вход

Программа всегда ожидает четыре поля:

1. `sourceBytes` - 16 байтов в hex;
2. `aMapping` - 16 индексов для сборки вектора `a`;
3. `bMapping` - 16 индексов для сборки вектора `b`;
4. `keyBytes` - 16 байтов ключа в hex. [^input-parse]

CLI-скрипт умеет получить эти данные тремя способами:

1. из JSON-файла через `--payload-json`;
2. из отдельных аргументов `--source-bytes`, `--a-mapping`, `--b-mapping`, `--key-bytes`;
3. интерактивно через `input()`, если аргументы не переданы. [^generate-input]

После этого создается объект `payload`, который и уходит в `run_trace(payload)`. [^generate-main]

## Что происходит сразу после запуска расчета

Функция `run_trace()` начинает с проверки входа:

- `parse_hex_vector()` проверяет, что `sourceBytes` и `keyBytes` содержат ровно 16 hex-байтов;
- `parse_mapping()` проверяет, что `aMapping` и `bMapping` содержат ровно 16 целых индексов от 0 до 15. [^input-parse]

После валидации программа создает первый служебный блок `actions` и кладет туда исходные таблицы для `source` и `key`. Каждая таблица строится функцией `vector_df()`, которая показывает для каждого байта индекс, hex, binary и polynomial.

## Как программа собирает рабочие векторы `a` и `b`

Следующий шаг - построение двух 16-байтовых векторов из исходного массива `source`.

Это делает функция `mapping_df()`:

- берет индекс из `aMapping` или `bMapping`;
- читает соответствующий байт из `source`;
- добавляет этот байт в выходной вектор;
- одновременно записывает строку в таблицу соответствия.

На этом этапе программа получает:

- `a_vector`;
- `b_vector`;
- две таблицы, которые потом показываются в отчете.

## Как считается `X[k](a)`

Для операции `X[k](a)` вызывается `xor_trace(left, right)`.

Функция идет по двум 16-байтовым массивам парами и для каждого индекса:

1. берет байт из ключа;
2. берет байт из вектора `a`;
3. делает XOR;
4. записывает строку в таблицу со значениями `left_hex`, `right_hex`, `xor_hex`, `xor_bin`, `xor_poly`.

Результат возвращается как новый 16-байтовый массив `x_result` и как DataFrame для отчета. Затем `run_trace()` добавляет этот шаг в общий список `actions`.

## Как считается `S` и `S^-1`

Для подстановки используется `s_trace(arr, inverse=False)`.

Она работает так:

1. выбирает таблицу `PI` для прямого преобразования или `PI_INV` для обратного;
2. проходит по всем 16 байтам входного массива;
3. заменяет каждый байт по таблице;
4. записывает в таблицу вход и выход в hex, binary и polynomial.

В `run_trace()` эта функция вызывается дважды:

- отдельно для `S(a)`;
- отдельно для `S^-1(a)`. 

Отдельно внутри шага `F[k](a, b)` подстановка снова применяется, но уже не к `a`, а к результату `X[k](a)`.

## Как считается умножение в `GF(2^8)`

Внутри линейных шагов используется побитовое умножение в поле `GF(2^8)`.

Для подробной трассировки работает `gf_mul_trace(a, b)`:

1. запускает цикл на 8 итераций;
2. на каждой итерации проверяет младший бит второго множителя;
3. при необходимости XOR-ит текущий множимый в аккумулятор;
4. сдвигает множимое влево;
5. если был старший бит, делает редукцию через `MOD_POLY_LOW`;
6. сдвигает второй множитель вправо;
7. записывает все промежуточные значения в таблицу. [^gf-mul]

Для тех мест, где подробная таблица не нужна, используется сокращенный вариант `gf_mul_fast(a, b)`. Он делает ту же операцию, но без накопления промежуточной таблицы.

## Как считается линейная функция `l(a15..a0)`

Функция `l_trace(arr, include_mul_bit_steps)` берет 16 байтов входного массива и 16 фиксированных коэффициентов `L_COEFS`.

Дальше она делает два прохода:

1. для каждой пары `(coef, value)` вызывает `gf_mul_trace()` и получает одно произведение;
2. затем последовательно XOR-ит все 16 произведений в аккумулятор `acc`.

На выходе `l_trace()` возвращает:

- итоговый байт `acc`;
- блоки с таблицами;
- вложенные шаги с детальной побитовой трассировкой умножений, если `include_mul_bit_steps=True`.

## Как считается `R(a)` и `R^-1(a)`

Для `R(a)` вызывается `r_step(arr, include_mul_bit_steps)`.

Внутри этой функции:

1. считается `l_value = l(a15..a0)`;
2. создается новый массив `result = [l_value] + arr[:15]`;
3. строится таблица перестановки байтов;
4. возвращается и сам новый массив, и готовое дерево шагов.

Для `R^-1(a)` используется `r_inv_step(arr, include_mul_bit_steps)`.

Она работает по обратной схеме:

1. собирает `shifted = arr[1:] + [arr[0]]`;
2. считает `l(shifted)`;
3. строит новый массив `result = arr[1:] + [l_value]`;
4. формирует таблицу перестановки для отчета. 

## Как считается `L(a)` и `L^-1(a)`

Полная линейная операция считается не отдельной формулой, а повторением шага `R`.

В коде это сделано двумя уровнями:

- `l_fast()` и `l_inv_fast()` просто 16 раз подряд вызывают `r_fast()` или `r_inv_fast()`;
- `l_action()` делает то же самое, но еще собирает историю раундов и блоки для отчета.

То есть программа на каждом раунде хранит:

- состояние до шага;
- состояние после шага;
- таблицу истории всех 16 раундов. [^l-action]

## Как считается `F[k](a, b)`

Шаг `F[k](a, b)` полностью собирается в `f_action(k, a, b)`.

Порядок работы внутри функции такой:

1. `xor_trace(k, a)` строит `X[k](a)`;
2. `s_trace(x_result)` строит `S(X)`;
3. `l_action(s_result)` строит `L(S(X))`;
4. `xor_trace(l_result, b)` строит первый выход `F.first`;
5. второй выход просто копируется как `a[:]`.

Функция возвращает:

- кортеж `(first_result, second_result)`;
- готовое дерево блоков для отображения шага `F` в интерфейсе и в отчете.

## Что именно возвращает `run_trace()`

После расчета всех шагов `run_trace()` она собирает структурированный результат:

- `actions` - дерево шагов и подшагов;
- `summaryTableHtml` - финальная таблица по ключевым результатам;
- `checksTableHtml` - таблица проверок обратимости.

Внутри этой же функции в конце выполняются проверки:

- `S^-1(S(a)) == a`;
- `R^-1(R(a)) == a`;
- `L^-1(L(a)) == a`.

После этого результат готов либо для API-ответа, либо для сохранения в отчет.
