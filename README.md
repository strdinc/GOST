# GOST / Kuznyechik Web Lab

Учебный проект для лабораторных работ по криптографии. Приложение **пошагово** показывает, как работают преобразования шифра «Кузнечик» и арифметика в поле GF(2^8), включая промежуточные таблицы, двоичные представления и полиномиальные формы.

> Проект ориентирован на прозрачность вычислений для проверки вручную, а не на производительность.

---

## 1) Что делает программа

Программа принимает:

- исходный вектор `source` из 16 байтов (`b0..b15`),
- две перестановки индексов для построения векторов `a` и `b`,
- ключ `k` из 16 байтов.

После этого строит подробную трассировку:

1. построение `a` и `b` из `source`;
2. `X[k](a) = k XOR a`;
3. `S(a)` и `S^-1(a)`;
4. `R(a)` и `R^-1(a)`;
5. `L(a)=R^16(a)` и `L^-1(a)`;
6. шаг Фейстеля `F[k](a,b) = (LSX[k](a) XOR b, a)`;
7. проверки обратимости (`S^-1(S(a))=a`, `R^-1(R(a))=a`, `L^-1(L(a))=a`);
8. финальную сводную таблицу результатов.

Каждая операция отображается в виде HTML-таблиц (`pandas.DataFrame`) и разворачиваемых шагов в интерфейсе.


### Поток данных

1. Пользователь вводит данные на frontend.
2. Frontend отправляет `POST /api/run`.
3. Flask вызывает `run_trace(payload)`.
4. Сервис возвращает JSON со структурой шагов (`actions`) и HTML сводной таблицы.
5. Frontend отрисовывает шаги как дерево раскрывающихся блоков.

## 3) Форматы входных данных

### 3.1 `sourceBytes`

- 16 шестнадцатеричных байтов;
- допустимы форматы вида `55`, `0x55`, регистр не важен.

Пример:

```text
55 65 51 33 4D 95 59 C7 93 8C BD E3 D6 AB 2F 79
```

### 3.2 `aMapping` и `bMapping`

- по 16 целых индексов в диапазоне `0..15`;
- индекс указывает, какой элемент `source` взять в текущую позицию.

Пример:

```text
aMapping: 3 13 14 11 4 10 15 9 1 0 12 2 5 8 6 7
bMapping: 15 7 9 0 6 12 2 14 10 4 8 1 11 3 13 5
```

### 3.3 `keyBytes`

- 16 шестнадцатеричных байтов.

Пример:

```text
88 99 AA BB CC DD EE FF 00 11 22 33 44 55 66 77
```

## 4) Backend (Flask) — API и поведение

Файл: `backend/app.py`.

### Эндпоинты

- `GET /api/health` — проверка доступности API (`{"status":"ok"}`).
- `POST /api/run` — основной расчет трассировки.
- `GET /` и `GET /<path>` — раздача SPA-сборки frontend из `backend/dist`.

### Обработка ошибок

- Если тело `POST /api/run` не JSON → `400`.
- Ошибки валидации входа (например, не 16 байтов) → `400`.
- Непредвиденная ошибка в вычислениях → `500`.
- Если frontend не собран (`backend/dist/index.html` отсутствует) → `500` с подсказкой команды сборки.

## 5) Криптологический сервис: объяснение всех функций

Файл: `backend/src/services/trace_service.py`.

Ниже — функции в логических группах с назначением.

### 5.1 Константы и таблицы

- `MOD_POLY_FULL = 0x1C3` — полный вид неприводимого многочлена GF(2^8).
- `MOD_POLY_LOW = 0xC3` — «нижняя часть» для редукции при сдвиге.
- `PI` — S-box Кузнечика (256 значений).
- `PI_INV` — обратный S-box, вычисляется автоматически.
- `L_COEFS` — коэффициенты линейного функционала `l(...)` для R-преобразования.

### 5.2 Вспомогательные функции представления

- `b8(value)` — байт в виде двоичной строки `xxxxxxxx`.
- `poly_str(value, width=8)` — полиномиальная форма над GF(2), например `x^7 + x^4 + 1`.
- `bytes_hex_str(arr)` — вектор байтов как `AA BB CC ...`.
- `bytes_bin_str(arr)` — вектор байтов в двоичном виде.

### 5.3 Парсинг и валидация входа

- `_normalize_tokens(value, field_name)` — унифицирует строку/массив в список токенов.
- `parse_hex_byte(token)` — проверяет и переводит hex-токен в `int`.
- `parse_hex_vector(value, field_name)` — валидирует вектор из **ровно 16** байтов.
- `parse_mapping(value, field_name)` — валидирует 16 индексов `0..15`.

### 5.4 Формирование блоков для UI

- `block_text(text)` — создает текстовый HTML-блок.
- `df_to_html(df)` — рендер `DataFrame` в HTML-таблицу.
- `block_table(df, caption=None)` — табличный HTML-блок с подписью.
- `node(title, blocks=None, steps=None)` — узел дерева шагов (заголовок, блоки, вложенные шаги).

### 5.5 Табличные представления векторов/операций

- `vector_df(arr)` — таблица: индекс, hex, bin, poly для каждого байта.
- `mapping_df(source, mapping, output_name)` — применяет перестановку и строит таблицу `output_i = source_j`.
- `xor_trace(left, right)` — побайтовый XOR + подробная таблица (hex/bin/poly).
- `s_trace(arr, inverse=False)` — S-box/обратный S-box по каждому байту.

### 5.6 Арифметика GF(2^8)

- `gf_mul_trace(a, b)` — **пошаговое** умножение в поле:
  - контроль LSB множителя,
  - условный XOR в аккумулятор,
  - сдвиг,
  - редукция по `MOD_POLY_LOW`.
  Возвращает результат и таблицу побитовых шагов.
- `gf_mul_fast(a, b)` — компактная версия без подробной таблицы (для быстрых проверок).

### 5.7 R/L преобразования

- `l_trace(arr, include_mul_bit_steps)` — вычисляет `l(a15..a0)` как XOR 16 произведений `coef_i * a_i`.
- `r_step(arr, include_mul_bit_steps)` — выполняет `R(a)` и возвращает результат + шаги.
- `r_inv_step(arr, include_mul_bit_steps)` — выполняет `R^-1(a)` + шаги.
- `r_fast(arr)` / `r_inv_fast(arr)` — быстрые версии без детальной трассировки.
- `l_fast(arr)` / `l_inv_fast(arr)` — 16-кратное применение `R` / `R^-1`.
- `l_action(arr, inverse=False)` — полный сценарий `L` или `L^-1` с историей 16 раундов.

### 5.8 S и Feistel

- `s_fast(arr, inverse=False)` — быстрый S/обратный S без таблицы.
- `f_action(k, a, b)` — шаг Фейстеля:
  1. `X[k](a)`
  2. `S(X)`
  3. `L(S(X))`
  4. XOR с `b`
  5. второй выход равен исходному `a`

### 5.9 Главная функция расчета

- `run_trace(payload)` — центральный orchestrator:
  - парсит и валидирует вход;
  - выполняет все операции;
  - формирует список действий `actions` (дерево шагов);
  - добавляет таблицу проверок обратимости;
  - формирует финальную сводку `summaryTableHtml`.

Именно эту функцию вызывает API `/api/run` и CLI-скрипт `main.py`.

## 6) Frontend (React) — объяснение компонентов

Файл: `frontend/src/App.jsx`.

### Компоненты

- `HtmlBlock` — безопасно вставляет готовые HTML-блоки, пришедшие с backend.
- `TraceNode` — рекурсивный компонент для отображения дерева шагов (`details/summary`).
- `App` — основной экран:
  - хранит поля ввода,
  - запускает `runTrace()` (POST в `/api/run`),
  - отображает ошибки,
  - выводит финальную сводку и детальные действия.

### Важное по UI

- Все таблицы рисуются backend и передаются как HTML.
- На верхнем уровне показывается статистика: число действий и вложенных шагов.
- Корневые шаги открыты по умолчанию, вложенные можно разворачивать вручную.

## 7) CLI-режим (без браузера)

Файл: `main.py`.

Позволяет преподавателю/студенту сгенерировать автономный HTML-отчет из терминала.

### Как работает

1. Запрашивает 4 входа (`sourceBytes`, `aMapping`, `bMapping`, `keyBytes`).
2. Вызывает `run_trace(payload)` напрямую.
3. Создает:
   - `trace_report.html` — полный отчет с шагами,
   - `trace_payload.json` — сохраненный входной набор.

### Запуск

```bash
python main.py
```

## 11) Что проверять преподавателю на защите

- Студент понимает, как строятся `a` и `b` из `source`.
- Может объяснить, где и зачем используется `MOD_POLY_LOW = 0xC3`.
- Показывает таблицу шагов `gf_mul_trace` и объясняет редукцию полинома.
- Может пройти путь `X -> S -> L -> XOR b` в `f_action`.
- Показывает проверки обратимости и объясняет их смысл.
- Умеет сопоставить hex/bin/poly представление одного и того же байта.
